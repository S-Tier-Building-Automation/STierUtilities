S-Tier Utilities is a **capability platform**. Every tool is described by a manifest and declares the capabilities it **provides** and **requires**. A kernel resolves that dependency graph at boot and brokers a scoped service bus, so any app or connector can reuse another tool's functionality through a stable, versioned contract — without reaching into its internals.

## The model in one minute

- **Tools** are manifest-driven. A manifest has an `id`, `version`, `category` (`service` or `app`), the capabilities it `provides`, the ones it `requires` (with a semver range), and the `permissions` it needs.
- **Capabilities** are the unit of reuse — a named, versioned interface (e.g. `bacnet.read@1.0`). Whoever provides it (compiled-in, a local process, or a remote MCP server) is invisible to consumers.
- **The kernel** validates manifests, resolves `requires` against providers by semver, computes a boot order, and hands each tool a *scoped* host that can only reach the capabilities it declared.
- **Consuming** a capability is one call: `const svc = host.use("netscan.v1")`.

## Why versioned contracts

Because consumers depend on `capability@^major.minor`, a provider can be swapped or reimplemented without breaking anyone, and multiple major versions can coexist during a migration. The [capability reference](/reference/overview) is generated directly from the running platform, so it never drifts from the code.
