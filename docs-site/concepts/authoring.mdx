A tool is a manifest plus its implementation. The manifest is the single source of truth — the kernel boots from it, and the [capability reference](/reference/overview) is generated from it.

## The manifest

```js
{
  id: "my-tool",            // kebab-case, unique
  name: "My Tool",
  version: "1.0.0",         // semver — the tool's own version
  apiVersion: "1",          // platform manifest API version
  kind: "native",           // native | mcp | webview
  category: "app",          // app | service (drives the docs grouping)
  provides: [               // capabilities this tool exposes to others
    { capability: "my.capability", version: "1.0" },
  ],
  requires: [               // capabilities this tool consumes
    { capability: "timeseries", version: "^1.0", optional: true },
  ],
  permissions: ["fs.appdata"],
}
```

Capability versions in `provides` are **contract** versions, independent of the tool's own `version`. Bump a provided capability's major only on a breaking interface change so consumers' semver ranges keep resolving.

## Consume a capability

Declare it in `requires`, then resolve it from your scoped host. The host only resolves capabilities you declared (least privilege):

```js
// host.use throws if the capability is undeclared or unavailable.
const ts = host.use("timeseries.v1");
ts.write({ measurement: "my_metric", fields: { value: 42 } });

// host.tryUse returns null instead of throwing — use it for optional deps.
const netscan = host.tryUse("netscan.v1");
if (netscan) await netscan.scan("192.168.1.0/24");
```

Mark a dependency `optional: true` when your tool degrades gracefully without it. Required dependencies that can't be resolved are a boot-time error.

## Provide a capability

List it in `provides`, then register an implementation during setup. The implementation is the stable interface other tools call — keep it free of UI concerns:

```js
host.provide("my.capability", "1.0", {
  doThing: (arg) => /* ... */,
});
```

A **service** (`category: "service"`) typically owns an implementation and runs headless; an **app** is the user-facing surface. A capability can be extracted out of an app into a service so any tool can reuse it — see how `bacnet.read` is owned by the headless BACnet service in the [reference](/reference/bacnet-read).

## Permissions

Declare every permission your tool needs in `permissions`; the broker grants the declared subset and your code asserts them via `host.requirePermission(...)`. Permissions a *provider* holds are listed on each capability's reference page so consumers know what they're leaning on.

## Third-party tools (MCP)

A `kind: "mcp"` tool provides its capabilities by proxying to an out-of-process MCP server — consumers can't tell whether a capability is compiled-in or remote. That's the same swappability that lets any provider satisfy a contract.
